# Work Log · July 16, 2025 (Thursday)

**Hours:** ~4

## Overview

Day 3 of the i18n push: the seven tool pages. Yesterday made the site's chrome bilingual;
today every tool page followed — labels, placeholders, hints, button text and, the
interesting part, error messages that originate deep inside library code.

## Today's Work

### 1. The Design Problem: Errors Born in Library Code

Most of today was mechanical extraction, but the calculator posed a real design question.
Its expression evaluator (`lib/calc.ts`) throws errors like "unbalanced parentheses" —
and until today it threw them as **hardcoded Chinese prose**. The library can't call
translation hooks; it shouldn't even know languages exist.

The fix was to make the library throw a **key, not a sentence**: a `CalcError` class
carrying a machine-readable key (`parens`, `badNumber`, `unknownName`…) plus any values
(like the offending character). The calculator page catches it and maps the key into the
active locale via the `calculatorPage.errors` namespace, falling back to a generic message
for anything unexpected. Separation restored: the library reports *what* went wrong, the UI
decides *how to say it*.

The JSON tool got the same treatment in miniature — its line/column error helper now
returns coordinates instead of a formatted sentence, and the page builds the message with
`t('errorAt', {line, column})`.

### 2. Two Details That Only Show Up When You Translate

- **ICU braces.** The JSON placeholder contains a literal example object — but `{` and `}`
  are syntax in ICU messages. Instead of fighting the escaping rules, I pass the example in
  as a parameter: the message says `…e.g. {example}` and the code supplies the literal
  JSON string. Cleaner than escape gymnastics.
- **Locale-aware dates.** The timezone tool was formatting the comparison table with a
  hardcoded `zh-CN` formatter. Now it reads the active locale (`useLocale()`), so English
  readers get `07/16/2025, 14:30:00` where Chinese readers get `2025/07/16 14:30:00` —
  same instant, right convention.

Also caught the last hidden corner: the shared **CopyButton**'s "复制 / 已复制" labels,
which I'd walked past a dozen times without seeing.

### 3. Verification, and a Lesson About Verifying

After lint and a green build (19 static pages), I served the site and grepped each page for
expected strings — and every check passed suspiciously hard: `/en/base64` "contained" the
calculator's strings too. The catch: next-intl embeds the **entire message catalogue** in
each page's payload, so a naive grep matches the dictionary, not the UI. Re-ran the checks
against the **visible HTML only** (scripts stripped): 12 of 14 passed, and the two
"failures" were the timezone comparison table, which by design only renders after mount
(the hydration-safe `Date.now()` from last week). Verified its per-locale placeholders
instead. A test that can't fail isn't a test — worth remembering.

Final sweep: `grep` for any CJK character across `app/`, `components/` and `lib/` — zero
hardcoded Chinese left in code. Every user-facing string now lives in the catalogues.

## Outcome

All seven tools are fully bilingual, including their error messages. The site is now 100%
message-driven: adding a third language would be a single JSON file, no code changes. The
lasting lesson is the error-key pattern — library code should throw identifiers, not prose,
and today's refactor made that concrete.
