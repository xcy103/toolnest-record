## Today's Work

### 1. Learning Base64 (morning)

Started by learning what Base64 actually is and why it exists. The core idea: it's a way to represent arbitrary binary data (bytes) using only 64 "safe" text characters — `A–Z`, `a–z`, `0–9`, `+` and `/` — so that data can travel through channels that only reliably handle plain text. Worked through the mechanics: the input bytes are read in groups of 3 (24 bits), which are then split into 4 groups of 6 bits, and each 6-bit value (0–63) maps to one Base64 character. When the input length isn't a multiple of 3, the output is padded with `=` characters. That's also why Base64 output is always about 33% larger than the original.

One important point I made sure to understand: **Base64 is encoding, not encryption** — it provides no security at all, anyone can decode it. Its real everyday uses are things like embedding small images or fonts directly in web pages via `data:` URIs, and carrying binary data inside text-only formats like JSON or email.

### 2. How to Implement Base64 in the Front-end (morning)

Looked into how to actually do this in the browser. The built-in functions are `btoa()` (encode) and `atob()` (decode), but I learned they have a well-known trap: they only work on "binary strings" where every character is in the 0–255 range, so passing in text with Chinese characters or emoji throws an error. The fix is to bridge through **UTF-8 bytes**:

- **Encode:** use `TextEncoder` to turn the text into a UTF-8 byte array, convert those bytes into a binary string, then call `btoa()` on it.
- **Decode:** call `atob()` to get the binary string back, read it into a byte array, then use `TextDecoder` to turn the UTF-8 bytes back into readable text.

I also learned that `TextDecoder` can run in a strict "fatal" mode that throws when the bytes aren't valid UTF-8 — which turns out to be a clean way to validate that a pasted string is really valid Base64, instead of silently producing garbage.

### 3. Building the Base64 Tool (evening)

Built the tool page (`/base64`) using the concepts from the morning:

- Encode / decode mode switch, with the current result carried over when switching so it's easy to round-trip.
- Full UTF-8 support (Chinese, emoji, etc.) using the `TextEncoder` / `TextDecoder` approach above.
- Input validation with friendly error messages — an invalid Base64 string shows a clear prompt instead of crashing — plus tolerance for whitespace and line breaks in pasted input.
- A copy button on the output and a clear button on the input.

I also pulled the common page structure (breadcrumb, title, panels) and the copy button into shared components, so the next tools can reuse them instead of repeating the same markup.
