## Today's Work

### 1. Learning Dates & Timezones in the Browser (morning)

Started with the mental model. The key realization: a moment in time is really just **one number** — milliseconds since the Unix epoch (1970-01-01 UTC), the same everywhere on Earth. A "timezone" isn't part of that number; it's only a lens for *displaying* it. So the whole tool is: keep one epoch value as the source of truth, and render it through different lenses.

Things I learned about the JS `Date` object and its sharp edges:

- `Date.now()` gives epoch milliseconds; `new Date(ms)` wraps it. Unix timestamps in the wild are usually in **seconds**, so I need to multiply/divide by 1000 and let the user pick the unit.
- `new Date("2026-07-10T09:00:00")` (no zone suffix) is parsed as **local** time, while `.toISOString()` always prints **UTC** with a `Z`. Mixing these up is the classic off-by-hours bug.
- The right tool for showing a moment in an arbitrary timezone is **`Intl.DateTimeFormat`** with a `timeZone` option (an IANA name like `Asia/Shanghai`). It handles DST and offsets correctly, so I don't hand-roll any offset math. `Intl.DateTimeFormat().resolvedOptions().timeZone` also tells me the user's own timezone.

### 2. The Hydration Trap (morning)

A subtle one specific to this being a server-rendered site: I can't initialize state to `Date.now()` during render, because the server renders one time and the browser hydrates at a slightly different time, causing a hydration mismatch (and it wouldn't be static-prerenderable). The fix is to start with an empty/null value and set "now" in an effect that only runs on the client. Learned to render neutral placeholders until that first client tick.

### 3. Building the Timestamp / Timezone Tool (evening)

Built the tool page (`/timezone`):

- **One epoch value as source of truth**, edited from either side: a Unix timestamp field (with a seconds/milliseconds toggle) and a `datetime-local` picker, kept in sync so converting either way is instant.
- A "use current time" button, an ISO 8601 line with a copy button, and input validation (integers only, friendly errors) so half-typed input doesn't blow up.
- A **multi-timezone table** — UTC, Shanghai, Tokyo, London, New York, Los Angeles — all showing the selected moment at once via `Intl.DateTimeFormat`, which is genuinely useful when coordinating across the time difference between here and home.

Reused the shared components again.

### 4. Lint Lesson, Continued

Hit the same React rule as yesterday — "avoid setState directly in an effect" — this time on the legitimate "initialize to now on mount" pattern. Applied the same fix I learned: wrap the state updates in a small named function called inside the effect, so the effect body itself doesn't call setState directly. Lint went green without weakening the behavior.
