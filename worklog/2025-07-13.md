## Today's Work

### 1. A Decision: First Dependency (learning / planning)

Every tool so far has been zero-dependency on purpose — I even hand-wrote MD5 and a full
expression parser rather than pull in a package. A QR code encoder is a different beast:
it involves data encoding modes, Reed-Solomon error correction, bit-matrix placement, and
mask selection. Writing that from scratch correctly is a multi-day project on its own.

At the same time, I've decided the next phase (Chinese/English i18n) will use **next-intl**,
which means the project is taking on a dependency anyway. So the zero-dep rule has served
its purpose — it kept the early tools simple and forced me to actually learn the
algorithms — but it's no longer worth stretching to breaking point. For QR I chose the
mature, widely-used **`qrcode`** library. Pragmatism over purity, now that the codebase is
established.

### 2. Learning How QR Codes Work (at a high level)

Even using a library, I read up on the model so the options I expose make sense:

- A QR code encodes data plus **error-correction** data, so it can still be read when
  partially damaged or obscured. There are four levels — **L, M, Q, H** — recovering
  roughly 7%, 15%, 25% and 30% of the code respectively. Higher levels are more robust
  (e.g. a logo overlaid in the center) but hold less data at a given size.
- The library picks the smallest "version" (grid size) that fits the data at the chosen
  error-correction level, which is why very long input can fail to generate — so that
  needed a friendly error message.

### 3. Building the QR Tool (build)

Built the tool page (`/qrcode`):

- A text/URL input, an **error-correction level** selector (L/M/Q/H) and a **size**
  selector (256/320/512).
- Generates the code with `QRCode.toDataURL(...)` into a PNG data URL, rendered as an
  image with a **download-as-PNG** button (the data URL doubles as the download href).
- Async generation handled with the same `cancelled` flag pattern from the hash tool, so
  fast edits don't render a stale code, plus a friendly error when the content is too long.

Everything still runs 100% in the browser — the library generates the code client-side, so
no data leaves the machine, consistent with the rest of ToolNest.
