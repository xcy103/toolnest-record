## Overview

The final day of the i18n work. The routing, the translated chrome and the translated tool

## Today's Work

### 1. The Language Switcher

The switcher looks trivial but has one requirement that makes it more than a link: it
should change language **without leaving the current page**. Someone reading
`/zh/calculator` who clicks "EN" wants `/en/calculator`, not to be dumped back on the home
page.

next-intl makes this clean once you know the pieces: `usePathname()` returns the path
**without** the locale prefix (`/calculator`), and `router.replace(pathname, { locale })`
re-renders that same path under the chosen locale — and updates the locale cookie so the
choice sticks on the next visit. I confirmed those signatures from the type definitions
before writing anything, since this is exactly the kind of API that's easy to assume wrong.

I built it as a small two-button segmented control (EN / 中文) matching the existing theme
toggle's style, with `aria-current` marking the active locale and the whole group labelled
for screen readers. Wrapped the navigation in `useTransition` so the switch doesn't block
the UI. It reuses the same idea as the theme toggle — a small client control in the navbar —
so it sits naturally next to it.

### 2. Bilingual Sitemap + robots

For search engines to treat `/en/base64` and `/zh/base64` as translations rather than
duplicate content, the sitemap has to spell out the relationship. Read the sitemap file
convention for this version of Next and used `alternates.languages`: each page appears once,
keyed by the default locale, with `hreflang` links for `en`, `zh`, and `x-default`
(pointing at English, the default). Generated it from the tool registry so it can never
drift from the actual list of tools. Added a matching `robots.ts` that points crawlers at
the sitemap.

One thing I had to get right: the base URL. There's no custom domain yet, so I made it an
env var (`NEXT_PUBLIC_SITE_URL`) with a Vercel fallback — when the domain is ready, it's a
one-line change, not a code hunt.

### 3. A Debugging Detour (and a Real Lesson)

The build suddenly failed with "Could not locate request configuration module" — next-intl
couldn't find `i18n/request.ts`, a file I hadn't touched in days. Five minutes of confusion
until I checked `pwd`: earlier I'd stepped into `node_modules/.../docs` to read the Next
docs, and my shell was still there, so `next build` was running from the wrong directory.
The code was fine. Lesson: when something that worked yesterday breaks "impossibly," check
the environment before the code.

### 4. Verification

Served the production build and checked the real output:

- `/sitemap.xml` — 8 URLs (home + 7 tools), each with `en` / `zh` / `x-default` hreflang
  alternates ✅
- `/robots.txt` — allows crawling, points to the sitemap ✅
- Language switcher renders in both locales with the correct active state ✅
- `/sitemap.xml` and `/robots.txt` are correctly *excluded* from the locale redirect (the
  proxy matcher skips paths with a file extension) ✅

The build now emits 21 static pages: every tool in both languages, plus the sitemap and
robots routes.

## Outcome

The internationalization arc is complete. ToolNest serves fully translated English and
Chinese from `/en` and `/zh`, visitors can switch between them from any page without losing
their place, and search engines are told the two are the same site in two languages. What
started as "is there a plugin for this?" ended as a proper i18n setup — locale routing,
message catalogues, locale-aware navigation, localized metadata, a switcher, and a
bilingual sitemap.
