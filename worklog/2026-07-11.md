## Today's Work

### 1. Why Not `eval`? (morning)

The lazy way to build a calculator is `eval(expression)`, but I learned why that's a bad idea: `eval` runs *arbitrary* JavaScript, so a tool that evals user input is a security hole, and it also wouldn't give me control over things like degrees vs radians or factorial. Since ToolNest runs everything in the browser on user input, I wanted a real, sandboxed evaluator I fully understand. So the morning was spent learning how to parse and evaluate math expressions properly.

### 2. Learning the Shunting-Yard Algorithm (morning)

Learned the classic three-stage pipeline for evaluating expressions:

1. **Tokenize** — scan the string into tokens: numbers, operators, parentheses, function names, constants.
2. **Shunting-yard** (Dijkstra's algorithm) — convert the tokens from normal "infix" order into **Reverse Polish Notation (RPN)**, using a stack and each operator's *precedence* and *associativity* to decide ordering. This is where `2+3*4` becomes `2 3 4 * +`.
3. **Evaluate the RPN** — walk the RPN with a stack: push numbers, and when you hit an operator pop its operands, compute, push the result.

I also learned the fiddly parts that make or break correctness:
- **Precedence & associativity**: `^` is right-associative (`2^3^2` = `2^(3^2)` = 512), while `+ - * /` are left-associative.
- **Unary minus** is a different operator from subtraction, decided by what came before it. Getting `-3^2` to equal `-9` (not `9`) took real care — the negation has to bind *looser* than the power, and as a prefix operator it shouldn't pop anything off the stack on the way in. I verified the tricky cases (`-3^2`, `2^-2`, `-3!`) against known answers.
- **Implicit multiplication**: people write `2π` and `3(4+1)`, so I add an extra pass that inserts a `*` between adjacent values, which makes the calculator feel natural.

### 3. Building the Evaluator + UI (evening)

Put the evaluator in `lib/calc.ts` (tokenize → implicit-multiply → RPN → evaluate) with functions (sin/cos/tan and inverses, sqrt, ln, log, exp, abs, floor…), constants (pi/e/tau), factorial, and a degrees/radians flag for trig. Every failure throws a friendly Chinese message (unbalanced parens, unknown name, incomplete expression, divide-by-zero).

Before writing a single line of UI I **tested the evaluator against ~20 expressions** in Node, including the nasty precedence cases and the error cases, and fixed two bugs the tests caught (the `-3^2` sign and treating `1/0`/Infinity as an error). Only then built the page (`/calculator`):

- A scientific button pad (digits, operators, functions, constants) plus a directly-editable expression field, so you can click or type.
- **Live evaluation** as you type, with a degrees/radians toggle, a copy button on the result, and AC / backspace controls.
- Result formatting that trims floating-point noise (so `0.1+0.2` reads as `0.3`, not `0.30000000000000004`).

### 4. Version Control & Push

Verified with eslint + a production build (`/calculator` prerenders static), committed — `feat: add scientific calculator tool` — and pushed to `toolnest`.
