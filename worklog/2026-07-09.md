## Today's Work

### 1. Learning Hashing & the Web Crypto API (morning)

Started with what a cryptographic hash function actually is: a one-way function that turns any input into a fixed-size "fingerprint" (the digest). The same input always gives the same digest, a tiny change to the input scrambles the whole output, and you can't go backwards from the digest to the input. Common everyday uses are file checksums, deduplication, and integrity verification — *not* the same thing as encryption, since there's no "decrypt".

Then I learned the browser's built-in **Web Crypto API**: `crypto.subtle.digest(algorithm, bytes)`. It takes the algorithm name and a byte buffer and returns a Promise of an `ArrayBuffer`, which I convert to a hex string. Two things I had to internalize:

- It's **asynchronous** (returns a Promise), unlike the synchronous string functions in the earlier tools. So the tool needs to compute in an effect and handle the input changing while a previous computation is still in flight.
- It supports **SHA-1, SHA-256, SHA-384, SHA-512 — but deliberately not MD5**, because MD5 is considered broken for security and the standard chose not to encourage it.

### 2. The MD5 Problem (morning → design decision)

The home-page card promised MD5, but Web Crypto won't do it and ToolNest has a rule of staying pure front-end with no backend. Two options: pull in an npm package, or implement MD5 myself. Since the project currently has *zero* third-party runtime dependencies (only Next and React) and I wanted to keep that clean, I chose to implement MD5 directly.

That meant actually learning the MD5 algorithm: pad the message to a multiple of 512 bits (append a `1` bit, then zeros, then the original length as a 64-bit little-endian number), then process each 512-bit block through 64 rounds of bitwise operations over four 32-bit state words, using precomputed sine-derived constants. I wrote it to operate on a UTF-8 byte array so Chinese and emoji hash correctly. Crucially, I **verified it against known test vectors** — `md5("")`, `md5("abc")`, the classic "quick brown fox" — plus a multi-block input, before trusting it. All matched.

### 3. Building the Hash Tool (evening)

Built the tool page (`/hash`):

- One text input, and **all four digests computed at once** (MD5 from my implementation, the three SHA variants from Web Crypto), each on its own row with its own copy button — more useful than a dropdown you have to toggle.
- Handled the async correctly with a `cancelled` flag so a fast typist doesn't get a stale digest from an earlier keystroke.
- A note under the output warning that MD5 and SHA-1 are not suitable for security use (like password storage) and should be limited to checksums / fingerprints — I didn't want the tool to quietly imply they're safe.

Reused the shared components again. The MD5 code went into `lib/md5.ts` rather than the page, keeping the page focused on UI.

### 4. A Lint Lesson

Hit a new React lint rule — "avoid calling setState directly within an effect" — from my first draft that reset state at the top of the effect. Learned the cleaner pattern: move all the state updates *inside* the async function the effect calls, including the empty-input reset, so the effect body itself never calls setState directly. Fixed it, lint went green.
