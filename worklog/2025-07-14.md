# Work Log · July 14, 2025 (Tuesday)

**Hours:** ~4

## Overview

First day of Part 2: making ToolNest bilingual. Today was an **infrastructure day** — the
goal was explicitly *no visible change*. Get locale routing working underneath the site
(`/en`, `/zh`), with English as the default, and leave the actual translation for the

turned out to matter more than expected.

## Today's Work

### 1. Learning: How Next.js 16 Does i18n (morning)

Read the official internationalization guide for the version I'm actually on rather than
assuming. Two things stood out:

- **The routing model.** Locales live in a dynamic `[locale]` segment, so every page moves
  under `app/[locale]/`, and the locale becomes a param forwarded to every layout and page.
  Combined with `generateStaticParams`, Next pre-renders a full copy of the site *per
  locale* at build time — which is exactly what I want for SEO: `/en/base64` and
  `/zh/base64` are separate, independently indexable URLs.
- **`middleware` is now `proxy`.** Next.js 16 deprecated the `middleware` file convention
  and renamed it to `proxy` (there's even a codemod). This was the day's real trap: nearly
  every i18n tutorial online tells you to create `middleware.ts`, which is the *old* name.

### 2. Choosing the Library, and a Compatibility Check

I picked **next-intl** for locale routing. Before committing to it I checked its
`peerDependencies` — it explicitly allows `next: ^16.0.0`, so it's current. But since it
still ships its handler as `next-intl/middleware`, I had to work out how that meets the new
`proxy` convention. Reading the types answered it: `createMiddleware(routing)` returns a
plain `(request) => response` function, and the proxy docs say the file just needs to
export a single function. So next-intl's handler drops straight into `proxy.ts` as a
default export. Verifying that from the types instead of guessing saved me from the
"copy the tutorial, wonder why nothing redirects" trap.

### 3. Building the Infrastructure (evening)

- `i18n/routing.ts` — locales `['en', 'zh']`, **default `en`** (the whole point is to be
  open to the world, so `/` should land on English).
- `i18n/request.ts` — resolves the locale per request and loads its message catalogue,
  falling back to the default for unknown values (the `[locale]` segment acts as a
  catch-all, so `/unknown.txt` shouldn't crash).
- `i18n/navigation.ts` — locale-aware `Link` / router wrappers.
- `proxy.ts` — the locale redirect, using Next 16's new convention.
- Wrapped `next.config.ts` with the next-intl plugin.
- Moved every route under `app/[locale]/` (with `git mv`, so history follows the files) and
  turned the root layout into a locale layout: `<html lang={locale}>`, a `hasLocale` guard
  that 404s on junk, `setRequestLocale` for static rendering, and `generateStaticParams`
  so both locales pre-render.

### 4. The Bug I Nearly Shipped

The build went green, but green isn't working. Internal links still used `next/link` with
hrefs like `/base64` — no locale. A reader on `/zh/base64` clicking "home" would fall out
of Chinese. Swapped `Link` in Navbar, ToolCard and ToolLayout for the locale-aware one from
`i18n/navigation`, then **verified it by actually fetching the pages**: `/zh` now links to
`/zh/calculator`, `/en` links to `/en/calculator`. Lesson: a passing build says the code
compiles, not that the feature works.

### 5. Verification

Ran the production server and checked real requests rather than trusting the build table:

- `/` → **307 redirect to `/en`** ✅ (English default, as designed)
- `/en`, `/zh/base64`, `/en/qrcode` → all 200 ✅
- `<html lang>` correctly renders `en` / `zh` per locale ✅
- Bonus: next-intl emits **`hreflang` alternate-link headers** automatically — a Day 5 SEO
  task that came free ✅

The build now pre-renders 19 static pages: every tool, in both languages.

Also fixed a pre-existing lint error in `ThemeToggle` (the same "no setState directly in an
effect" rule I hit twice last week), so `npm run lint` is green across the project again.

## Outcome

ToolNest is now served from `/en` and `/zh`, both fully pre-rendered, with `/` redirecting
to English — and the site looks exactly the same as yesterday, which was the goal. The
groundwork for translation is in place. The most valuable lesson was concrete: this version
of Next renamed middleware to proxy, and reading the actual docs and type definitions for
*my* version beat following any tutorial.
